# REST API

The Clinical Image Anonymizer REST API provides a local HTTP interface for running image anonymization workflows programmatically.

This API is intended for local-first academic and research workflows. It reuses the same anonymization core used by the GUI and CLI.

> **Important safety note:** un-anonymized clinical images may contain sensitive information. Do not send un-anonymized images over a network unless you have explicitly verified that the network, machine, access control, institutional approvals, and privacy safeguards are appropriate. Network use is the user's responsibility.

---

## Supported status in v1.0.0

| Mode | Status | Recommendation |
|---|---|---|
| Localhost, `127.0.0.1` | Supported | Recommended default |
| Local network, `0.0.0.0` | Technically possible | Experimental, use only on trusted networks |
| Public internet | Not supported | Do not expose |
| Clinical production | Not supported | Research prototype only |
| Authentication | Not included | Add before shared/network deployment |
| HTTPS/TLS | Not included | Add before shared/network deployment |

## Start the API locally

From the project root:

```powershell
cd C:\Users\rga580\KronoX-Projects\clinical-image-anonymizer

python -m uvicorn src.api.app:app --host 127.0.0.1 --port 8000
```

Then open:

```text
http://127.0.0.1:8000/docs
```

The `/docs` page is generated by FastAPI and allows interactive testing of the API.

---

## Start the API on a local network

The API can also listen on all network interfaces:

```powershell
python -m uvicorn src.api.app:app --host 0.0.0.0 --port 8000
```

Another computer on the same network may then call the API using the server computer's IP address, for example:

```text
http://192.168.1.50:8000/health
```

This requires:

1. The server computer and client computer must be on the same reachable network.
2. Windows Firewall or institutional firewall rules must allow inbound traffic on the selected port.
3. The client must use the IP address of the computer running the API.
4. Input and output paths in requests are interpreted on the server computer, not on the client computer.
5. The network must be trusted and appropriate for handling un-anonymized clinical images.
6. The user is responsible for privacy, security, and institutional compliance.

For v1.0.0, LAN use should be considered experimental.

## Endpoints

### `GET /health`

Returns a lightweight service health check.

PowerShell:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/health
```

curl:

```bash
curl http://127.0.0.1:8000/health
```

Expected response:

```json
{
  "status": "ok",
  "service": "clinical-image-anonymizer-api"
}
```

### `POST /anonymize-batch`

Runs batch anonymization on images stored on the server computer.

The endpoint uses local filesystem paths. When calling this endpoint from another computer, `input_dir` and `output_dir` must still be valid paths on the machine running the API server.

#### Request fields

| Field | Type | Required | Description |
|---|---:|---:|---|
| `input_dir` | string | yes | File or folder path on the API server |
| `output_dir` | string | yes | Output folder path on the API server |
| `rectangles` | list | no | Rectangles to black out |
| `prefix` | string | no | Prefix for exported files |
| `randomize` | boolean | no | Randomize exported image order |
| `recursive` | boolean | no | Search input folders recursively |
| `workers` | integer | no | Parallel workers |
| `overwrite` | boolean | no | Allow overwrite of existing outputs |
| `preserve_structure` | boolean | no | Preserve input subfolder structure when recursive |

#### Rectangle format

Simple global rectangles can be sent as lists:

```json
[
  [10, 20, 220, 90],
  [250, 40, 480, 140]
]
```

Each rectangle is:

```text
[x, y, width, height]
```

These rectangles apply to all images.

Named rectangle objects can also be used:

```json
[
  {
    "filename": "image_001.png",
    "x": 10,
    "y": 20,
    "width": 220,
    "height": 90
  }
]
```

When `filename` is omitted, the rectangle is treated as global.

## PowerShell example

```powershell
$Body = @{
  input_dir = "C:\Users\rga580\KronoX-Projects\clinical-image-anonymizer\data\test_images"
  output_dir = "C:\Users\rga580\KronoX-Projects\clinical-image-anonymizer\outputs\api_test_batch"
  rectangles = @(
    @(10, 20, 220, 90),
    @(250, 40, 480, 140),
    @(40, 180, 180, 320)
  )
  prefix = "Api_"
  randomize = $true
  recursive = $false
  workers = 4
} | ConvertTo-Json -Depth 5

Invoke-RestMethod `
  -Uri http://127.0.0.1:8000/anonymize-batch `
  -Method Post `
  -ContentType "application/json" `
  -Body $Body
```

## LAN example

On the server computer:

```powershell
python -m uvicorn src.api.app:app --host 0.0.0.0 --port 8000
```

On another computer in the same trusted network:

```powershell
Invoke-RestMethod http://192.168.1.50:8000/health
```

Batch request from another computer:

```powershell
$Body = @{
  input_dir = "C:\ServerData\input_images"
  output_dir = "C:\ServerData\anonymized_output"
  rectangles = @(
    @(10, 20, 220, 90)
  )
  prefix = "Lan_"
  randomize = $false
  recursive = $false
  workers = 4
} | ConvertTo-Json -Depth 5

Invoke-RestMethod `
  -Uri http://192.168.1.50:8000/anonymize-batch `
  -Method Post `
  -ContentType "application/json" `
  -Body $Body
```

In this example, `C:\ServerData\input_images` and `C:\ServerData\anonymized_output` must exist or be valid on the server computer, not on the client computer.

## Privacy and security guidance

For v1.0.0:

- Prefer localhost use.
- Do not expose the API to the public internet.
- Do not send un-anonymized clinical images over an untrusted network.
- Do not run the API on shared networks without institutional approval and appropriate controls.
- Add authentication, HTTPS/TLS, access logging policy, and deployment hardening before any shared or production deployment.
- Users remain responsible for verifying that exported images are properly anonymized before sharing or using them.

---

## Limitations in v1.0.0

- No authentication.
- No HTTPS/TLS.
- No user management.
- No upload endpoint for multipart image transfer.
- Path-based API only: input/output paths are local to the server.
- No job queue or background task tracking.
- No automatic cleanup of output folders.
- Not a medical device.
- Not for diagnosis or clinical decision-making.
